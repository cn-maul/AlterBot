# 功能总览

本文描述当前代码中已经实现的功能。`design/` 下的文件用于记录历史设计和审查过程，不作为当前功能清单。

## 管理界面

Gentry 提供响应式 Web 管理界面，包含以下页面：

- 仪表盘：查看监控器运行状态、类型、URL、基线状态、上次检查时间、更新数量和最近错误；可直接启动、暂停、编辑或删除监控器。
- 新增/编辑监控：配置基本信息、内容提取和通知账户，并在保存前验证配置。
- 监控详情：查看运行状态、检查耗时、下次检查时间和基线状态；支持立即检查、重建基线、调整通知账户、查看更新记录、当前价格快照和价格变动事件。
- 推送管理：管理全局通知开关和多个推送账户。
- 扫描规则：通过 URL 和关键词预扫描页面，将候选结果保存为可复用规则，并导入或导出规则库。

界面支持亮色和暗色模式，并针对桌面和窄屏设备调整布局。

## 监控类型

### 网页新增监控

网页新增监控使用 `presence` 策略，适用于公告、新闻、更新日志和商品上新列表。

- 首次成功检查只建立基线，不把页面已有内容当作新内容推送。
- 后续出现新的稳定条目标识时生成更新记录。
- 标题、链接、摘要、日期等字段可以通过 CSS 选择器提取。
- 更新历史按较新内容在前的顺序展示。

### 价格监控

价格监控使用 `field_transition` 策略，当前支持金额类型。

- 支持单商品详情页和商品列表页。
- 单商品页可以使用页面 URL 作为商品身份。
- 列表页必须选择稳定且唯一的身份字段，例如 SKU 或商品链接。
- “价格发生下降”支持任意降价、最低金额、最低百分比，以及同时满足金额和百分比。
- “降到目标价及以下”只在价格从目标价以上跨越到目标价或以下时通知。
- 无效价格不会覆盖最后一次有效价格基线，币种变化不会直接跨币种比较。

## 内容提取与预扫描

每个监控保存一份独立的提取配置：

- `container`：页面中的内容区域。
- `item`：区域中的重复条目。
- `fields`：从每个条目中提取的文本或属性字段。
- `transform`：字段清理和转换配置。

新增或编辑监控时可以输入关键词执行预扫描。扫描器会结合关键词位置、重复列表、链接簇、表格和已保存规则生成候选，并显示样本内容。候选卡片上会标注策略来源——规则命中显示红色 `规则「xxx」`，启发式策略显示策略类型（关键词定位、重复列表等）。用户确认候选后，生成的选择器会应用到监控表单；容器选择器、列表项和字段等手动配置折叠在"高级设置"中，可按需展开调整。

预扫描只是配置辅助工具，不会自动创建监控或修改已有监控。保存监控后，运行时使用的是该监控自身保存的选择器副本。

## 扫描规则库

扫描规则用于复用已经确认的页面结构。创建流程为：

1. 输入 URL 和可选关键词。
2. 执行预扫描并选择正确候选。
3. 输入规则名称。
4. 由用户选择适用范围并保存。

保存后的规则会优先参与后续预扫描的候选生成。删除规则不会删除监控，也不会改变已经保存到监控中的提取配置。

### 适用范围

| 范围 | 匹配方式 | 适合场景 |
| --- | --- | --- |
| 当前页面 `exact` | 规范化后主机、路径和查询参数完全一致 | 固定列表页；同域名下系统较多时的默认选择 |
| 当前路由 `route` | 主机相同，按路径段前缀匹配；源 URL 有查询参数时继续要求查询参数一致 | 同一路由及子路由使用相同页面框架 |
| 通用结构 `global` | 不限制 URL；只有页面实际存在规则选择器时才生成候选 | 多个网站使用同一套通用网站框架 |

范围不会由系统自动扩大。新规则默认使用“当前页面”，用户确认确实需要复用时再选择“当前路由”或“通用结构”。根路径且没有查询参数时不能创建路由规则，以免退化为整个域名匹配。

查询参数会按规范顺序保存。例如 `?b=2&a=1` 和 `?a=1&b=2` 视为同一页面，而不同参数值不会互相匹配。

### 导入与导出

- 规则库可以导出为版本化 JSON 文件，包含名称、适用范围、选择器、字段、状态和说明。
- 导入前会校验文件版本、CSS 选择器、标题字段和适用范围。
- 单次导入支持 1 到 500 条规则。
- 同名规则会跳过，不覆盖本地已有规则。
- 导入过程使用数据库事务，写入失败时不会保留不完整规则。

旧版本保存的 `URLContains` 规则继续兼容。它们在界面中标记为“旧版”，仍按原来的 URL 子串方式匹配。

## 监控生命周期与基线

- 监控器可以启动、暂停、编辑和删除。
- 支持从详情页立即执行一次检查。
- 新建监控、手动重置基线，或修改影响检测语义的配置后，会重新建立基线。
- 重建基线会清除当前比较基准，但不会删除历史事件。
- 修改名称、检查间隔或通知账户等非检测配置，不应重建基线。
- 监控快照、变化事件和配置版本保存在 SQLite 中，进程重启后可以继续比较。

## 通知

当前支持以下推送服务：

- PushPlus：Token 必填，可配置渠道和模板。
- Webhook：URL 必填，默认使用 `POST`。
- Server酱：SendKey 必填，可配置渠道。
- Bark：设备 Key 必填，可配置服务器、分组、提示音和图标。

通知能力包括：

- 全局通知开关。
- 多推送账户管理，敏感配置在接口响应中脱敏。
- 每个监控可以选择零个或多个推送账户。
- 可推送全部符合规则的变化，或仅推送包含指定关键词的变化。
- 事件和通知投递任务持久化，支持异步发送、失败重试、去重和终态记录。

## 数据与兼容性

SQLite 数据库保存：

- 监控定义和字段配置；
- 当前条目快照和配置版本；
- 变化事件与旧版新增记录；
- 通知账户和投递任务；
- 扫描规则及其字段；
- 系统设置。

数据库由 GORM 在启动时自动迁移。升级程序不会自动删除历史监控结果或扫描规则。生产环境仍应定期备份 `gentry.db` 或 Docker 数据卷。

## 访问限制

默认抓取器请求目标服务器直接返回的 HTML，适用于无需登录的公开网页。以下页面可能无法正常抓取：

- 必须执行 JavaScript 才能生成主体内容；
- 需要登录态、验证码或浏览器指纹；
- 带有 JavaScript 安全验证、WAF 或严格反爬策略；
- 仅在浏览器网络请求中返回数据。

这类网站需要无需验证的数据源或独立的浏览器渲染抓取适配器。当前静态抓取器不会绕过安全验证。
